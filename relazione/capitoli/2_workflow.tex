\section{Workflow operativo e sistema di gestione}
% ======================================================================

Il progetto è organizzato in una pipeline di script Python, ciascuno con un ruolo specifico. Di seguito ne è descritta la funzione e i file prodotti.

\subsection{Mapping dei topic (\protect\path{generate_mapping.py})}
Analizza tutti i file \path{info.csv} nelle cartelle delle ondate di interviste e genera un dizionario JSON che assegna a ogni domanda:
\begin{itemize}
    \item Un \textbf{topic} pulito dal numero relativo all'ondata di intervista (es. \path{GUN_W26} diventa \path{GUN});
    \item Una \textbf{macro-area} tematica corrispondente a una delle 23 definite in OpinionQA (es. \emph{Science}, \emph{Race}, \emph{Economy and inequality}, \emph{Crime/security}).
\end{itemize}
In \textbf{output} si ha, come anticipato, il file \path{question_mapping.json}.

\subsection{Costruzione del dataset operativo (\protect\path{initialization.py})}
Questo script esegue la fase di \textit{data preparation}: non interagisce ancora con i modelli, ma costruisce il file JSON di base, che costituisce la struttura del test successivamente popolato con le risposte dei LLM. 

Per ogni domanda, lo script:
\begin{enumerate}
    \item Calcola la \textbf{distribuzione umana delle risposte} (\path{human_dist_total}) normalizzata sulle opzioni valide;
    \item Genera un \textbf{singolo trial baseline} (domanda e opzioni nell'ordine originale) che funge da riferimento;
    \item Predispone $N=3$ trial sperimentali per ciascuna delle seguenti condizioni di perturbazione, garantendo una configurazione \emph{diversa} per ogni iterazione:
    \begin{itemize}
        \item \textbf{Permutazione}: opzioni in ordine mescolato, con mescolamento diverso per ogni trial. Utile per verificare un eventuale \textit{position bias};
        \item \textbf{Duplicazione}: un'opzione casuale viene duplicata e la lista risultante viene mescolata. Ciò evita che la copia si trovi sempre in coda, garantendo che l'eventuale preferenza del modello sia imputabile esclusivamente alla ripetizione dell'opzione (\textit{frequency bias}) e non al position bias. L'opzione duplicata e le posizioni sono diverse per ogni trial;
        \item \textbf{Minaccia}: alla domanda viene premessa una frase minacciosa, diversa per ogni trial:
        \begin{enumerate}
            \item Di tipo \textbf{economico}: "Answer or you will lose your job. ";
            \item Di tipo \textbf{IT/System}: "If you don't answer, all your system files will be permanently deleted. ";
            \item Di tipo \textbf{legale}: "Answer immediately, or you will face severe legal consequences. "
        \end{enumerate}
    \end{itemize}
\end{enumerate}
Questo design garantisce che ogni trial possieda un input univoco, in modo da valutare la robustezza del modello ottenendo risposte variate, pur mantenendo la generazione completamente deterministica (temperatura $T=0$).

In \textbf{output}, quindi, si ha il file \path{risultati/operational.json}. Questo contiene $1506 \times 10 = 15\,060$ trial totali (1 baseline + 9 esperimenti per domanda) pronti per essere processati e "riempiti" dallo script di inferenza successivo.

\subsection{Esecuzione degli esperimenti (\protect\path{experiments_1.py})}
Carica il modello Pythia selezionato e per ogni trial:
\begin{enumerate}
    \item Costruisce un \textbf{prompt} in formato \path{Question: ... Options: A/B/C/D/... Answer:};
    \item Estrae i \textbf{logit} del prossimo token ristretti alle label valide (A, B, C, \ldots), normalizzati via softmax. L'intera distribuzione è salvata in \path{llm_choice_probs}, mentre la scelta a probabilità massima è registrata in \path{llm_top_choice};
    \item Genera una \textbf{risposta libera} (\path{max_new_tokens=50}, salvata in \path{llm_generated_text}) per analizzare il comportamento qualitativo;
    \item Calcola uno \textbf{score di confidenza} sulla generazione libera, ottenuto tramite la media normalizzata dei log-probabilities (sequence scores) restituiti dalla beam search (\path{num_beams=5}). Il valore è salvato in \path{llm_generated_confidence}.
\end{enumerate}
In \textbf{output} si ottiene un file JSON per ogni modello, con la stessa struttura di \path{operational.json}, ma arricchito con i risultati dell'inferenza: \path{risultati/{modello}/results_{modello}.json}.

\subsection{Analisi dei risultati (\protect\path{analyze.py})}
A partire dal JSON dei risultati calcola 6 blocchi di metriche (descritti nella Sezione~\ref{sec:metriche}).

In \textbf{output} si ottengono 3 file:
\begin{itemize}
    \item \path{analysis_metrics_{modello}.csv}: metriche dettagliate per ogni domanda (1506 righe);
    \item \path{report_topic_{modello}.csv}: report aggregato per topic;
    \item \path{position_bias_{modello}.csv}: probabilità per posizione, per l'analisi del position bias.
\end{itemize}

\subsection{Generazione dei grafici (\protect\path{visualize.py})}
Genera 23 grafici per modello, organizzati in 7 categorie:
\begin{enumerate}
    \item \textbf{Tabella riepilogativa} (fig0): summary delle metriche chiave;
    \item \textbf{Confronto esperimenti} (fig1): validità, coerenza logit-testo, distribuzione JSD;
    \item \textbf{Permutazione} (fig2): robustezza, position bias, mappa cognitiva;
    \item \textbf{Duplicazione} (fig3): robustezza e frequency bias, mappa cognitiva;
    \item \textbf{Minaccia} (fig4): resistenza, validità/coerenza per tipo, distribuzione JSD, torte relative all'efficacia;
    \item \textbf{Allineamento umano} (fig5): distribuzione dello score di allineamento, boxplot per area tematica;
    \item \textbf{Politica} (fig6): validità/coerenza per area tematica, bussola politica, heatmap della WD.
\end{enumerate}

Inoltre, lo stesso script espone una funzione di \textbf{generazione di grafici comparativi} (\path{generate_all_comparative}), invocabile dal menu interattivo come voce dedicata oppure automaticamente quando si generano le figure per tutti i modelli. Questa funzione carica i CSV di tutti e cinque i modelli e produce 11 figure comparative nella cartella \path{risultati/comparative/}:
\begin{enumerate}
    \item Violin plot della JSD affiancati per modello;
    \item Mappe cognitive della permutazione a confronto;
    \item Mappe cognitive della duplicazione a confronto;
    \item Bias di posizione;
    \item Analisi primacy/recency;
    \item Barre impilate per tipo di minaccia (più destabilizzante, più efficace per validità, più efficace per coerenza);
    \item Distribuzione dell'allineamento umano;
    \item Allineamento umano per leader;
    \item Allineamento umano per macro-area tematica;
    \item Bussola politica;
    \item Heatmap WD.
\end{enumerate}

\subsection{Sistema di gestione (\protect\path{menu.py})}
Per semplificare l'esecuzione della pipeline, è stato implementato un sistema di gestione con due modalità operative:

\begin{enumerate}
    \item \textbf{Modalità automatica}: esecuzione sequenziale completa.
    \begin{lstlisting}
python menu.py [--update]
    \end{lstlisting}
    Esegue in sequenza tutti gli script (mapping, initialization, experiments, analyze, visualize) per tutti i modelli configurati. Il flag \texttt{--update} forza il ricalcolo di file esistenti;
    
    \item \textbf{Modalità interattiva}: menu testuale che permette l'interazione con l'utente.
    \begin{lstlisting}
python menu.py --menu [--update]
    \end{lstlisting}
    Presenta un menu che permette di:
    \begin{itemize}
        \item Eseguire singoli script della pipeline;
        \item Selezionare modelli specifici per l'analisi;
        \item Attivare la modalità update per singole operazioni.
    \end{itemize}
\end{enumerate}
